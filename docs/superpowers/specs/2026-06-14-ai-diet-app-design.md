# AIダイエットアプリ 設計書

- 日付: 2026-06-14
- ステータス: 承認済み（ブレインストーミング完了）
- リポジトリ: `diet-ai-app/`（独立リポジトリ・GitHub Pages 配信 / hinyan1016）

## 1. 目的とスコープ

食事の写真、またはコンビニ弁当などの栄養成分ラベル写真をAIで分析し、カロリー・栄養素を概算して、どう行動すればよいかを提案する**個人用のスマホPWA**。

- **対象ユーザー**: 自分専用（個人ツール）。プライバシー・医療免責の懸念は最小。
- **利用端末**: スマホ中心（外出先・食卓で「その場で」撮影）。ホーム画面に追加して使うPWA。
- **MVP機能（すべて含む）**:
  1. 写真 → 栄養分析 → 行動提案（核）
  2. その日の合計・履歴
  3. 目標設定と進捗
  4. コンビニ栄養成分ラベルの読み取り
  5. トレンド（週次）の助言

### 非スコープ（YAGNI）
- 一般公開・マルチユーザー・アカウント認証（将来拡張時に構成Bへ移行）
- サーバー側データ保存（データは端末内のみ）
- 旧iOS Safari互換（個人の現行端末前提）

## 2. アーキテクチャ（構成A：サーバーなし）

```
📱 スマホPWA（GitHub Pages・静的配信）
   └ APIキーは端末の localStorage に保存
        ↓ 写真を直接送信 / 結果(JSON)を受信
🤖 Claude API（Vision・Tool Use）
        ↓
💾 IndexedDB（端末内）食事ログ・履歴・目標・体重
```

- 追加インフラ不要。既存のGitHub Pages運用と同じ。
- 食事データはサーバーへ送らず端末内に保存。AI分析時のみ画像をAPIへ送信。
- 分析のたびに自分のAnthropic従量課金が発生。

## 3. AI連携（心臓部）

### 構造化出力（Tool Use）
`record_nutrition` ツールを1つ定義し、`tool_choice` で強制呼び出し。Claudeは必ず決まったJSONを返す。

返却スキーマ（例）:
```json
{
  "name": "幕の内弁当（推定）",
  "kcal": 680,
  "protein_g": 24,
  "fat_g": 19,
  "carb_g": 98,
  "salt_g": 3.2,
  "fiber_g": 4,
  "confidence": "high | mid | low",
  "items": ["白米", "焼き魚", "卵焼き", "煮物"],
  "advice": "炭水化物多め。夜は主食を半分にして..."
}
```

### ブラウザ直接呼び出し
- リクエストに `anthropic-dangerous-direct-browser-access: true` ヘッダを付与（CORS許可）。
- APIキーは localStorage から読み込み。リポジトリにハードコードしない。

### 2つの分析モード（同一ツール定義を使い分け）
| モード | 入力 | プロンプト方針 |
|---|---|---|
| 🍽 料理写真 | 食事写真 | 見た目から品目・量を推定し栄養を概算。確信度を返す |
| 🏷 ラベル写真 | 栄養成分表示の写真 | 記載値を読み取り転記。推定ではなく**記載値優先** |

撮影画面でトグル選択。中身は同じ `record_nutrition` に集約。

### 行動提案の3レイヤー
1. **単発提案**: 写真単体の結果（`advice` フィールド）
2. **1日視点の提案**: 記録後、「今日の合計＋目標」を渡して再生成
3. **週次トレンド助言**: 直近7日の集計を渡して傾向＋助言を生成

### モデル
- 既定: `claude-sonnet-4-6`（Vision精度とコストのバランス）
- 設定で `claude-opus-4-8` に切替可

## 4. データ構造（IndexedDB・すべて端末内）

```
meals            id, datetime, mode("photo"|"label"), imageThumb,
                 name, kcal, protein_g, fat_g, carb_g, salt_g, fiber_g,
                 confidence, items[], advice, userEdited(bool)
goals (1件)      kcal, protein_g, fat_g, carb_g, salt_g, weightTarget?
settings (1件)   apiKey, model, adviceTone
weights          date, kg   （任意）
```

- 画像は原寸保存せず**サムネのみ**保存（容量節約）。
- 合計・履歴・トレンドはすべて `meals` を日付集計して算出。
- 設定に **JSON書き出し/取り込み**（機種変更・バックアップ用）。

## 5. 画面構成（5タブ PWA）

| タブ | 内容 |
|---|---|
| 🏠 ホーム（きょう） | 今日の合計kcal・PFC進捗バー＋目標、記録した食事リスト |
| 📅 履歴 | 過去の日別記録（カレンダー/リスト） |
| 📷 撮る（中央・主役） | カメラ起動 → 料理/ラベルトグル → 分析中 → 結果（修正/記録） |
| 📈 トレンド | 週間グラフ＋傾向の助言 |
| ⚙️ 設定 | APIキー、モデル、目標、データ入出力 |

分析結果画面: 推定料理名・kcal・PFC・塩分＋💡行動提案＋「修正」「記録する」。

## 6. 技術スタック（ビルドなし）

ネイティブESモジュールでビルドツール不要。多数の小さいファイルに分割。

```
diet-ai-app/
  index.html              画面骨組み＋下タブ
  css/styles.css
  js/
    db.js                 IndexedDBラッパ
    ai.js                 Claude API呼び出し＋record_nutritionツール定義
    camera.js             撮影・画像縮小（サムネ生成）
    nutrition.js          集計・目標進捗・トレンド計算（純粋関数）
    ui.js                 画面描画・タブ切替
    app.js                起点
  manifest.json / sw.js   PWA（ホーム追加・オフライン履歴閲覧）
  icons/
```

- JS記法: `const`/`let`/`async-await`/モジュール（個人の現行端末前提）。
- グラフは依存を増やさず **SVG/Canvas で自前描画**。
- PWA: ホーム追加、オフラインで記録・履歴・トレンド閲覧可（AI分析のみネット必須）。

## 7. エラー処理（境界での検証）

- APIキー未設定 → 撮影前に設定画面へ誘導
- 通信失敗・レート制限 → わかりやすいメッセージ＋リトライ。撮影画像は破棄せず保持
- AI応答の**スキーマ検証**（数値型・必須フィールド）。壊れていれば再試行 or 手動入力へ
- `confidence: low` → 「確信度が低い」バッジ＋修正を促す
- オフライン → 記録・履歴・トレンド閲覧は可。AI分析のみネット必須と明示

## 8. テスト方針

- `nutrition.js` の集計・目標進捗・週次トレンド入力生成、AI応答スキーマ検証を **Vitest** でユニットテスト（AI応答はモック）。
- カメラ・実APIはモック境界まで。E2Eは個人ツールのため最小（手動確認）。
- 純粋関数を中心に据えることで、UIやAPIに依存せずロジックを検証可能。
